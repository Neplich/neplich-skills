# AI 对话助手 - 产品需求文档（PRD）

> 版本：v0.1 骨架  
> 状态：草稿 / 待评审

---

## 1. 项目背景与目标

### 1.1 一句话描述
一款面向个人用户或小型团队的轻量级 AI 对话客户端，采用经典“左列表、右聊天”的布局，帮助用户高效管理与 AI 的多轮会话。

### 1.2 目标用户
- **个人用户**：日常向 AI 提问、写作辅助、代码答疑。
- **轻度办公用户**：需要按主题拆分多个会话，避免上下文混乱。

### 1.3 核心目标
- 会话管理清晰：创建、切换、重命名、删除、搜索会话。
- 聊天体验流畅：多轮对话、Markdown 渲染、代码高亮、流式输出。
- 入口门槛低：单用户本地优先、无需登录即可使用（可后续扩展登录）。

## 3. 信息架构与页面结构

```
App
├── Left Sidebar（会话区）
│   ├── Header（Logo + 新建会话按钮）
│   ├── Search（会话搜索框）
│   └── Session List
│       └── Session Item（标题 + 最后更新时间 + 右键/ hover 操作菜单）
│
└── Right Main（聊天区）
    ├── Chat Header（当前会话标题 + 编辑 + 清空 + 删除）
    ├── Message List（可滚动区域）
    │   └── Message Bubble（用户 / AI）
    └── Chat Footer（输入框 + 发送按钮 + 中断按钮）
```

## 4. 交互流程

### 4.1 新建并发送第一条消息
1. 用户点击左侧“新建会话”按钮。
2. 列表顶部新增一条“未命名会话”，并自动切换到右侧聊天区。
3. 光标自动聚焦输入框。
4. 用户输入内容并回车发送。
5. 右侧聊天区先展示用户消息；随即向 AI 服务端发送请求。
6. AI 消息以流式方式出现在用户消息下方。
7. 会话列表中该条目标题自动更新为第一条消息的前 N 个字符（或由后端返回摘要）。

### 4.2 切换会话
1. 用户在左侧点击某一历史会话。
2. 右侧聊天区加载该会话的历史消息（若本地已缓存则直接渲染；若云端则需拉取）。
3. 输入框保持清空，光标不自动聚焦。

### 4.3 删除会话
1. 用户点击会话项旁的“更多”菜单，选择“删除”。
2. 弹出二次确认弹窗（或 inline 撤销条）。
3. 确认后移除该会话，右侧聊天区按以下规则切换：
   - 若被删会话之前还有其他会话，自动切换到时间最近的上一条会话；
   - 若列表为空，展示空状态占位页。

## 5. 界面原型（文字描述）

### 5.1 空状态
右侧聊天区在没有选中任何会话时显示：
- 居中图标 + 文案“新建一个会话开始对话吧”。
- 快捷“新建会话”按钮。

### 5.2 消息气泡
- **用户消息**：靠右对齐，背景色采用品牌主色，文字反白。
- **AI 消息**：靠左对齐，背景色与页面底色深浅区分，文字正文色。
- 气泡最大宽度约占聊天区 80%，避免过长行导致阅读困难。

### 5.3 代码块
- 代码块顶部显示语言类型标签与“一键复制”按钮。
- 使用等宽字体，背景与正文明显区分。

## 6. 数据模型（初步）

### Session（会话）
```ts
{
  id: string;           // 唯一标识
  title: string;        // 会话标题
  createdAt: number;    // 创建时间戳
  updatedAt: number;    // 最后活跃时间戳
  messages: Message[];  // 消息数组
}
```

### Message（消息）
```ts
{
  id: string;
  role: 'user' | 'assistant' | 'system';
  content: string;      // Markdown 文本
  createdAt: number;
  status?: 'sending' | 'streaming' | 'done' | 'error';
}
```

## 7. 技术约束与建议（不决定最终选型）

- **前端**：建议采用现代组件式框架开发，以便于状态管理与流式渲染。
- **本地存储**：若作为 Web 应用，可用 IndexedDB 或 localStorage 做轻量持久化；若作为桌面端，可用本地文件系统。
- **AI 接入**：优先支持标准 OpenAI 风格接口（/v1/chat/completions），方便后续切换不同模型供应商。
- **样式方案**：建议引入成熟的 UI 组件库或原子化 CSS 方案，加快左右分栏与响应式实现。

## 9. 待确认问题（Open Questions）

1. 目标平台是纯 Web（浏览器打开），还是桌面端应用（Electron/Tauri）？
2. AI 接口来源：是否自备 API Key 调用第三方大模型，还是有自部署的后端服务？
3. 是否需要多语言支持？若需要，默认语言是中文还是英文？
4. 是否需要消息的多选删除或批量导出？
5. 会话命名策略：取用户第一条消息前 20 字，还是由 AI 自动总结标题？
